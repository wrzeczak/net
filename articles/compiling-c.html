To compile C code on Windows, I've found there are generally two good toolsets to use; either Visual Studio, or a newer tool called w64devkit. I'll discuss each in some more detail below this, though I will note I don't currently use Visual Studio.

w64devkit


w64devkit is basically a replacement for the MINGW toolset, which is a toolset that tries to emulate the standard Linux toolset, on Windows. I've always found MINGW (and its relation CYGWIN) to be clunky, slow, and not really very functional; maybe I'm stupid, but I've never gotten it to work right on my machine.
w64devkit solves these problems by seemingly automagically setting up a MINGW toolset in a folder with a Docker image. I'd never used Docker before installing it, and I deleted Docker after installation (it doesn't depend on it to run, just to install, which is the great thing). The setup took about half an hour on my machine, which is standard according to the developers. I got it to work very easily in Visual Studio Code after adding it do my Windows PATH, and have been satisfied ever since. The only thing I've noticed so far is that Visual Studio's Intellisense doesn't really like it all that much, though I'm told that's a configuration error on my end. If you are looking to compile C code on Windows, this is probably your best option.

Visual Studio


Visual Studio is Microsoft's big development suite of applications and toolboxes. It's designed to run on Windows natively and is actually its own compiler, not gcc like w64devkit. It works fine, as it should. It's in my experience a little slow (though my machine isn't particularly powerful), especially Visual Studio, the application and code editor. It's frustratingly not very well integrated with library management, however. vcpkg is their library management tool, but not everything is on there and even then I've found it to be sometimes unreliable, especially outside of Visual Studio. However, I did like to use their GitHub integration when I was working on a C++ project with 2-3 other programmers. It was very clean and quick and I didn't need to understand how GitHub worked at all to start, which was nice until I hit a merge conflict and had no idea what to do (not Visual Studio's fault). This is a good option if you're looking for an integrated development environment, and if you're an eighth grader trying to collobrate with people with no clue what you're doing (like I was!).
